Grupos

Esta é uma atividade em GRUPO de até 3 pessoas. Neste sentido, é permitido que as pessoas do grupo discutam entre si livremente sobre a atividade em questão. Recomenda-se fazer o modelo antes de cada parte para que o grupo saiba a direção que o projeto está tomando. O projeto foi feito de forma que 2 ou 3 pessoas codifiquem em um tempo quase equivalente. Pode ficar sobrecarregado para fazer sozinho, mas não é algo impossível de ser feito. O mais complicado de fazer o projeto individualmente, é que você não terá com quem discutir / pedir ajuda.

A equipe do projeto deve estar especificada no arquivo README.md na raíz do projeto do git e deve ter um trecho no seguinte formato:

## Equipe

Nome do Aluno 1 - Matricula 1

Nome do Aluno 2 - Matricula 2

Nome do Aluno 3 - Matricula 3

Hospedagem de Código (Github)

Para o projeto, vocês devem hospedar o código no github e de forma privada. Crie um repositório privado e compartilhe com sua equipe e com o professor responsável (a ser definido). Cada pessoa deve ter sua conta configurada de forma que seja possível identificar o responsável por cada submissão (é MUITO importante ter configurado o git com git config --global user.name "Fulano" e git config --global user.email fulano@exemplo.com).

Caso as contribuições do aluno não apareçam no git, considera-se que o aluno não fez contribuições ao projeto.

Para noções iniciais sobre github, o monitor Gabriel Brandão, preparou um 2 vídeos curtos sobre o assunto (introdução | prática).

Responsabilidades

Apesar do projeto ser feito em grupo, cada aluno é responsável por uma parte bem definida. Isto é verificado a partir das contribuições e submissões do git. Um aluno pode subir pequenas correções ou modificações no código de outro, mas

O projeto é composto de 6 partes, sendo 3 liberadas agora e mais 3 liberadas na semana seguinte. Cada aluno irá desenvolver 1 ou mais partes, e testar 1 ou mais partes do projeto. O aluno não desenvolve a parte que testa, e vice-versa. Os grupos devem se organizar de forma que cada aluno seja responsável por uma quantidade equivalente de responsabilidades.

Como exemplo, considere o grupo de Pedro, Mariana e Alex:

  • Pedro pega a parte 1 e testa a parte 3
  • Mariana pega a parte 2 e testa a parte 1
  • Alex pega a parte 3 e testa a parte 2

Para grupos de 2 alunos, na primeira semana, o aluno pega 2 partes e testa uma terceira parte:

  • João pega parte 1 e 3, e testa parte 2
  • Joseph pega parte 2, e testa as partes 1 e 3
  • Depois, João Pega a parte 4, e testa parte 5 e 6
  • Joseph pega a parte 5 e 6, e testa a parte 4

Alunos codificando individualmente, codificam e testam todo o projeto.

A responsabilidade de cada membro, deve estar descrita no README.md do git no seguinte formato:

### Testes

Nome do Aluno 3 - 2, 6

Design e Testes

Para esse sistema é obrigatório usar o pacote "sapo" como base.

Dentro do pacote "sapo" deve existir a classe Facade que terá acesso a todos os controllers e operações do sistema. Essa classe segue o padrão de design Facade que centraliza o acesso a todas as operações e subsistemas do projeto.

Você é livre para importar ou fazer uso de qualquer classe do Facade, desde que siga duas regras:

  • A Facade deve estar no pacote sapo
  • Qualquer outro pacote do sistema deve estar dentro do pacote sapo
  • Todas as operações definidas no projeto devem estar na Facade, seguindo os mesmos nomes, parâmetros e retorno
  • O Facade deve ter apenas um construtor e que não recebe parâmetros

package sapo;

public class Facade {

    private PessoaController pessoaController;

    // … demais controllers

    public Facade() {

        this.pessoaController = new PessoaController();

        // … demais controllers

    }

    public void cadastrarPessoa(String cpf, String nome, String[] habilidades) {

        this.pessoaController.cadastrarPessoa(cpf, nome, habilidades)

    }

    // demais métodos …

}

Neste projeto, os alunos não são obrigados a fazer testes de unidades de todas as classes, mas são obrigados a desenvolver testes de sistema que testem as operações especificadas neste projeto e usem apenas a Facade para isso. No entanto, ainda recomendamos fazer testes de unidades, especialmente das classes mais complexas: isto facilita identificar problemas pontuais, mesmo que exista o teste feito com a Facade.

Especificação

O Sistema de Acompanhamento de Pessoas da Organização (SAPO) realiza operações de acompanhamento de pessoas e suas atividades em uma organização de ensino. O SAPO representa um sistema de informação CRUD. É um sistema de informação no sentido de um conjunto de partes que têm funções específicas mas, que em conjunto, tratam de um conjunto de dados. E é um CRUD pois realiza as operações de criação (Create), recuperação (Recover), atualização (Update) e apagamento (Delete) de dados.

Nesse sistema as pessoas são identificadas pelo seu CPF. As atividades representam um macro-objetivo a ser atingido. Cada atividade é composta por tarefas, onde uma tarefa representa uma ação específica de duração limitada dentro de uma atividade. Uma ou mais pessoas podem ser associadas a uma tarefa e cada tarefa pode ser parte de uma única atividade.

Para esse sistema, começaremos o desenvolvimento com 3 partes bem definidas:

  •         Cadastro de pessoas
  •         Cadastro de atividades
  •         Cadastro de tarefas

Durante todo sistema, caso uma identificação inválida seja passada para um método que deseja receber uma identificação exata (não confundir com termos de busca), deve ser lançada a exceção NoSuchElementException. Caso um atributo inválido seja passado, como strings vazias, deve ser lançada a exceção IllegalArgumentException. Caso se tente fazer uma alteração de estado de uma entidade do sistema, deve ser lançada a exceção IllegalStateException.

Pessoas

As pessoas do nosso sistema são identificadas unicamente pelo seu CPF. Além do CPF, cada pessoa recebe um nome e um array textual das habilidades que cada pessoa possui. O nome e o CPF não podem ser strings vazias durante o cadastro, mas o sistema pode receber uma lista vazia de habilidades.


Nesse sistema, deve ser possível exibir uma pessoa a partir do CPF. A representação textual de pessoa recém-cadastrada no sistema segue o modelo descrito abaixo contendo o nome da pessoa e seu CPF, além da lista de habilidades em ordem alfabética.


Matheus Gaudencio do Rêgo – 111.111.111-11
- Desenvolvimento web
- Professor
- Programador


É possível alterar o nome ou habilidades de uma pessoa, mas não deve ser possível inserir um nome em branco. As novas habilidades (inclusive nenhuma habilidade) substituem as habilidades atuais da pessoa. Respeitando a Lei Geral de Proteção de Dados (LGPD) deve ser possível remover uma pessoa do sistema. Ao remover uma pessoa do sistema, todas as suas associações devem ser igualmente removidas.

Além do cadastro e busca por pessoas, deve ser possível adicionar observações/comentários sobre a pessoa. Cada comentário tem a data atual no sistema, bem como a descrição do comentário inserido e o autor (CPF). Para isso, deve ser possível inserir comentários no sistema, bem como recuperar os comentários sobre uma pessoa na ordem que foram inseridas. Os comentários não aparecem na representação textual de uma pessoa, mas tem a seguinte representação:

Matheus Gaudencio do Rêgo – 111.111.111-11
Comentários:
-- É uma boa pessoa, só não tem muito juízo (Lívia Maria)
-- Bebe café demais! (Reinaldo)

Atividades

Uma atividade representa um conjunto de ações (tarefas) que devem ser tomadas para atingir um determinado objetivo. Exemplo, é possível ter como atividade, “Estudar OO” que pode ter como tarefas: “Revisar encapsulamento”, “Preparar material de estudo”, “Fazer apresentação sobre atributos e métodos”, etc. Por enquanto, as tarefas não serão modeladas. Por fim, cada atividade tem um responsável.

Cada atividade é cadastrada com um nome, descrição e responsável (CPF de uma pessoa), não vazios e não nulos. Para cada atividade é gerado um código único, formado pelas 3 primeiras consoantes do nome da atividade mais um número que representa a quantidade de atividades cadastradas até o momento. Atividades não podem ser removidas, mas podem ser desativadas. Na ausência de consoantes para gerar o ID, a consoante X é usada. (ex.: Code AOE vira CDX-4). Esse ID é retornado no cadastro.

A atividade:“Estudar OO”, se for a primeira a ser criada no sistema, recebe a identificação STD-0. Já a atividade “Revisar encapsulamento” se cadastrada em sequência, recebe o código RVS-1. E sucessivamente.

Uma atividade pode ser encerrada (concluída) ou desativada (abandonada ou invalidada). Só é possível encerrar ou desativar atividades sem tarefas pendentes. Uma atividade encerrada ou desativada não pode mais receber tarefas, mas essa atividade pode ser reaberta. Lance uma exceção caso o usuário tente alterar o estado da atividade de forma inválida (ex.: definir como encerrada uma atividade que foi desativada, ou reabrir uma atividade já aberta).

Uma atividade pode ser exibida a partir de seu ID, onde uma atividade é representada pelo seu nome, código, descrição, responsável, total de tarefas realizadas e cadastradas, e uma lista com as 3 primeiras tarefas pendentes (da mais recém-cadastrada para a mais antiga). Veja o exemplo abaixo:

STD-0: Estudar OO
Responsável: Matheus Gaudencio do Rêgo – 111.111.111-11
===
Atividade de estudo de OO, considerando alunos com experiência de programação e uso da linguagem Java.
===
Tarefas: 6/20
- Fazer apresentação sobre atributos e métodos - STD-0-20
- Preparar material de estudo - STD-0-15
- Revisar questões da prova - STD-0-10

Deve ser possível alterar a descrição ou responsável da atividade. Uma atividade pode não ter responsável (pessoa removida), Neste caso a linha do responsável não aparece em sua exibição.

Tarefas

As tarefas representam ações pontuais sendo realizadas em uma atividade. Cada tarefa tem uma duração (total de horas gastas até o momento), além de nome e habilidades recomendadas. Cada tarefa existe associada a uma única atividade e pode ter uma ou mais pessoas associadas.

Cada tarefa recebe um código que é composto pelo código da atividade além da ordem em que a tarefa foi cadastrada (independente se tarefas foram removidas ou não). Exemplo: STD-0-0, STD-0-1, STD-0-2, etc. Este código é retornado no cadastro da tarefa. Observe que a partir do código de tarefa, é possível obter o código da atividade na qual ela faz parte.

É possível alterar o nome e habilidades exigidas em uma tarefa, bem como acrescentar ou remover horas. É possível remover a própria tarefa do sistema, bem como exibir uma tarefa. Ainda, deve ser possível concluir uma tarefa. Uma tarefa concluída não pode ter suas horas ou responsáveis alterados.

A tarefa pode ser exibida a partir de seu código, e, ao ser exibida, apresenta todos os dados a ela associada, incluindo a atividade à qual ela pertence, a descrição, habilidades e a equipe de pessoas responsáveis (sem ordem definida). Exemplo:

Preparar material de estudo - STD-0-15
- Estudar OO
Professor, Objetos
(8 hora(s) executada(s))
===
Equipe:
Ingrid Xpto – 222.2222.222-22
Matheus Gaudencio do Rêgo – 111.111.111-11

Por fim, deve ser possível associar uma pessoa a uma tarefa (responsável), bem como remover essa associação.

Buscas

Nosso sistema agora precisa de um sistema de busca mais avançado para pessoas, atividades e tarefas.

Para pessoas, a busca de termos acontece, ao mesmo tempo, em CPF, nome e habilidade (maiúsculas e minúsculas são indistintas na busca). A busca deve retornar uma lista de pessoas, em ordem alfabética do nome (ou seja, NÃO é ordem lexicográfica), em que a chave de busca esteja contida no CPF, no nome da pessoa, ou que sejam uma das habilidades. Assim a busca por “Matheus Java” pode retornar a pessoa Matheus Gaudencio do Rêgo, caso ela tenha a habilidade Java. A busca por “Ensino Java” por sua vez pode retornar Lívia, Matheus e Reinaldo, desde que os 3 tenham as habilidades “Ensino” e “Java”.

Já as atividades são buscadas a partir de uma ou mais palavras de busca, e deve-se buscar tais palavras no nome da atividade, na parte textual do seu código, no código completo, no número do código e na descrição da atividade. Assim, é válido buscar: “RVS encapsulamento”, ou “RVS-1”, ou “1 encapsulamento”. Novamente, não é relevante se as chaves de busca estão em maiúsculas ou não. A busca retorna uma lista de representações com cada atividade, ordenadas pelo ID.

Por fim, tarefas podem ser buscadas a partir do nome da tarefa (ignorando a diferença entre maiúsculas e minúsculas), que retornará uma lista de representações textuais, onde cada elemento é a versão textual de uma tarefa que satisfaça a busca, da mais recém-cadastrada até a mais antiga. Deve ser possível buscar limitando a busca em uma atividade.

Ainda em tarefas, deve ser possível também buscar sugestões de tarefas para uma pessoa. As tarefas retornadas devem ser aquelas que não foram concluídas / abandonadas. As tarefas são ordenadas da seguinte forma:

  • das que têm mais habilidades em comum com a pessoa, até as que têm menos habilidades (ou seja, a pessoa e tarefa não precisam ter todas as habilidades em comum)
  • em caso de empate, das que não tem pessoas associadas primeiro,
  • e, em caso de empate, a partir do código, em ordem alfabética.

O sistema deve armazenar o histórico de buscas. Ao consultar a n-ésima busca mais recentemente realizada, o usuário deve retornar por padrão um array onde a primeira linha indica o tipo de busca feita (PESSOA, ATIVIDADE, TAREFA ou SUGESTÃO), e os demais elementos no array representam os elementos encontrados na busca.

+ buscarAtividade(consulta: str): str[]

Funções

  • IMPORTANTE: O responsável pela parte de PESSOA (Parte 1) não deve ser responsável por Funções

As pessoas de nosso sistema podem assumir diferentes funções. Algumas pessoas serão professores, outras serão alunos e algumas não terão uma função bem definida. As funções são úteis para acompanhar o crescimento de cada pessoa a partir das atividades realizadas.

Além de cadastrar pessoas, deve ser possível, imediatamente, cadastrar alunos e professores. Além disso, deve ser possível alterar a função associada a uma pessoa. Cada função carrega alguns detalhes, como exemplo, a função de aluno contém detalhes da matrícula e período. Já a função professor tem detalhes das disciplinas oferecidas pelo professor, bem como seu código SIAPE.

A partir do momento que uma pessoa é cadastrada como professor ou aluno, sua representação passa a exibir também essa informação. No caso de ser cadastro como aluno, aparece a função (Aluno) bem como a matrícula e período:

Tarefas Gerenciais

  • IMPORTANTE: O responsável pela parte de TAREFA (Parte 3) não deve ser responsável pela Tarefa Gerencial.

Nem toda tarefa é igual. Neste sentido, deve ser possível cadastrar uma tarefa especial, conhecida como "tarefa gerencial". Uma tarefa gerencial pode agrupar outras tarefas de forma que o tempo de realização dessa tarefa seja referente ao tempo gasto para gerenciar as demais tarefas.

Uma tarefa gerencial tem um nome e as habilidades requeridas ou recomendadas são definidas pela união de todas as habilidades das tarefas que a compõem, além da habilidade de "gestão". A identificação de uma tarefa gerencial segue o mesmo padrão de uma tarefa comum.

Deve ser possível adicionar e remover horas de forma exclusiva a essa tarefa gerencial. No entanto, a atividade é concluída se todas as tarefas que a compõem também forem concluídas. Nesta situação, não deve ser possível adicionar ou remover pessoas de uma tarefa gerencial concluída. Caso uma tarefa gerencial seja concluída, todas as tarefas associadas a ela também são concluídas. Caso uma tarefa gerencial seja removida, ela não afeta as tarefas associadas à tarefa gerencial.

É possível remover tarefas (em andamento ou concluídas) ou adicionar tarefas em andamento a uma tarefa gerencial, inclusive, adicionar tarefas gerenciais. Não deve ser possível adicionar uma tarefa gerencial que gere um ciclo (ex.: a tarefa gerencial TG1 contém a tarefa gerencial TG2, e a TG1 é adicionada a TG2 -- ciclo detectado).

As tarefas gerenciais podem ser buscadas e sugeridas Cada atividade gerencial conta como duas tarefas para a contagem de níveis das pessoas. A exibição de uma tarefa gerencial informa, além da equipe, as tarefas adicionadas (da mais recente para a mais antiga).
